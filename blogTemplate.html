HTMLCollection vs NodeList:
An HTMLCollection is an array-like data structure that holds elements. It is a live collection and provides methods to grab things from the list. Similarly, a NodeList is an array-like collection of Nodes that can be live or static and has methods to iterate over the list.
The main difference is that a NodeList holds nodes and an HTMLCollection just elements. Ex: querySelectorAll() returns a static NodeList while getElementsByTageName returns a live HTMLCollection

Mixins are a flexible way to distribute reusable functionalities for Vue components. A mixin object can contain any component options. When a component uses a mixin, all options in the mixin will be “mixed” into the component’s own options.
A mixin runs first before component's code, then it is bound (merged) by Vue to the component instance so it's easy to access mixin from component/instance scope, but not vice versa.

Using require() to read json files presents a caveat.
ex: const data = require('./sample.json');
require() will always cache the content of the loaded module (or file, in this case).
The next time require() is called again, it will restore it from the cache instead of reading it again.
This can be good or bad depending on your requirements.

Touch events:
For touch events such as touchstart, touchmove, touchend,event.clientX, event.clientYwill not work.
Touches is a read-only TouchList listing all the Touch objects for touch points that are currently in contact with the touch surface, regardless of whether or not they've changed or what their target element was at touchstart time.
let touches = touchEvent.touches
Ex: for mouse events you use event.clientX, but for touch events you use event.touches[0].clientX

Vue refs & lifecycle hooks

I had a problem calling the function findLatestData() in mounted(), which uses this.$refs.
$refswas in reference to an element using v-for. Even when created and mounted, a ref used in a v-for directive is not fully rendered. When ref is used together with v-for, the ref you get will be an array containing the child components mirroring the data source. In the Vue guide it says "$refs are only populated after the component has been rendered, and they are not reactive. It is only meant as an escape hatch for direct child manipulation - you should avoid accessing $refs from within templates or computed properties." Even using $nextTick()in mounted() I was having problems
I fixed my problem using beforeUpdate(), like so:
beforeUpdate() { let this1 = this; this.$nextTick().then(function() { this1.findLatestData() }) }

Reactivity in Vue.js

Generally only data and computed properties are reactive in Vue.
However, as I've discovered, methods can establish a reactive dependency. For example, in App.vue I refactored <router-links>to loop over data with a v-for, but I couldn't figure out a way to bind a class to data that referenced other data (computed properties). Here:
<router-link v-for="(routerLink, i) in routerLinks" :key="routerLink.name" :to="routerLink.link" tag="span" tabindex="0" class="hover" :class="{deactivated: getDeactivatedStatus(routerLink.deactivated)}">
Binded class deactivated is dependent on value of routerLink.deactivated, but routerLink.deactivated is simply a string representing a computed property obtained via Vuex store. So instead of just :class="{deactivated: routerLink.deactivated}, which would do nothing, we use
:class="{deactivated: getDeactivatedStatus(routerLink.deactivated)}
Our method getDeactivatedStatus returns our sought after computed property and IS reactive. Why is it reactive? Not because our method references a computed property, no. It is reactive because it gets called on each template render. And our template is rendered each time there is a change in login status.
